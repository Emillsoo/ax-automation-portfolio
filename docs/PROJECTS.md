# AX·업무자동화 프로젝트 사례집

## 사례 1. 지출결의서 RPA 자동화

### 프로젝트명

AI 입력 데이터와 그룹웨어를 연결한 지출결의서 작성 자동화

### 기간과 진행 상태

- 기간: 2025.11~2026.02 개발·개선
- 상태: 운영 중

### 배경과 기존 문제

담당자가 영수증 내용을 확인한 뒤 그룹웨어 지출결의서에 항목을 반복 입력해야 했다. 그룹웨어 화면은 iframe과 팝업을 사용하고, 계정과목과 승인선을 별도 화면에서 선택해야 해 단순 매크로만으로 안정적인 처리가 어려웠다.

### 요구사항

- 구조화된 지출 데이터를 그룹웨어 입력값으로 변환
- 계정과목·승인선·참조정보 자동 선택
- 작업 진행 상황을 사용자가 확인
- 실패 시 원인·단계·스크린샷 확인
- 작업 취소와 타임아웃 처리

### 담당 범위

외부 기반의 최소 골격을 바탕으로 핵심 기능 전체를 직접 구현했다. 앞단 영수증 OCR·구조화는 협력 시스템이 담당했고, 본인은 구조화된 데이터를 받은 이후의 API·작업 관리·화면 자동화·운영 기능을 개발했다.

### 시스템 구성과 데이터 흐름

```text
[협력 시스템의 영수증 구조화]
            ↓
      Flask REST API
            ↓
  작업 큐·상태·이력 관리
            ↓
 Robot Framework·Selenium
            ↓
 [그룹웨어] 화면 입력·저장
            ↓
 WebSocket 진행 상태·결과
```

### 기술 선택 이유

- Flask: 내부 자동화 호출용 API를 단순하게 구성
- Flask-SocketIO: 긴 RPA 작업의 진행 상태를 실시간 전달
- Robot Framework: 화면 동작을 업무 단계 단위 키워드로 분리
- Selenium: iframe·팝업이 있는 레거시 웹 화면 제어
- Redis 선택형 구성: 작업 이력을 유지하되 장애 시 메모리 방식으로 제한 운영

### 직접 구현한 내용

- 작업 시작·상태·결과·로그·취소 API
- Robot Framework 프로세스 실행과 변수파일 생성
- 표준 출력 파싱을 통한 업무 단계별 진행률 변환
- iframe 전환과 팝업 선택 키워드
- 계정과목·승인선·폼 데이터 매핑
- 타임아웃·프로세스 종료·실패 스크린샷
- 선택적 저장소 장애 시 메모리 폴백

### 기술적 어려움과 해결 과정

화면 구조가 iframe과 팝업으로 중첩되어 셀렉터가 쉽게 끊겼다. 화면 전환과 팝업 선택을 공통 키워드로 분리하고, 단계별 실패 위치를 기록했다. 승인선과 계정과목은 단순 문자열 입력이 아니라 검색·선택 과정이 필요해 데이터 매핑과 화면 동작을 분리했다.

### 보안 및 개인정보 보호

- 실제 계정과 URL은 환경설정에서 관리
- 외부 공개 시 `.env`, 로그, 스크린샷을 포함하지 않음
- 영수증·결의 데이터는 공개 데모에 사용하지 않음
- GitHub 재구현본은 로컬 모의 화면과 생성 데이터만 사용

### 배포·운영·장애 대응

Windows 서비스 환경을 고려해 프로세스 종료, 타임아웃, 로그 확인, 폴백을 설계했다. 자동화 성공 여부뿐 아니라 어느 업무 단계에서 멈췄는지 확인할 수 있도록 운영 가시성을 확보했다.

### 성과

반복적인 그룹웨어 입력을 자동화하고, AI 기반 문서 구조화와 RPA를 연결하는 처리 흐름을 구축했다. 사용자가 긴 작업의 상태를 확인하고 실패 원인을 추적할 수 있게 했다.

### 배운 점

RPA는 클릭 자동화만으로 완성되지 않으며 입력 계약, 작업 상태, 타임아웃, 실패 증거, 운영 프로세스가 함께 설계되어야 한다.

## 사례 2. 재원환자 불일치 자격확인 알림 자동화

### 프로젝트명

EMR 화면 업무 규칙을 재현한 재원환자 자격 불일치 자동 알림

### 기간과 진행 상태

- 기간: 2026.03~2026.06 개발·안정화
- 상태: 운영 중
- 최신 로컬 근거: v2.3 계열

### 배경과 기존 문제

입원 중에도 보험 자격이 바뀔 수 있어 담당자가 매일 EMR 불일치 화면을 확인했다. 확인이 늦어지면 수납·청구 업무에 영향을 줄 수 있었고, 단순 DB 비교만으로는 화면과 다른 결과가 나타날 수 있었다.

### 요구사항

- 원내 자격과 외부 자격 결과 자동 대조
- EMR 화면과 동일한 판단 기준 적용
- 실제 확인 대상만 업무 담당자에게 전달
- 대상 없음과 오류를 구분
- 날짜·기관·중복으로 인한 오탐 방지

### 담당 범위

현업 업무 분석, SQL, n8n 워크플로우, 데이터 변환, 시트·메일 연계, 운영 안정화를 직접 수행했다.

### 시스템 구성과 데이터 흐름

```text
n8n 스케줄
   ↓
[EMR 시스템] 읽기 전용 조회
   ↓
화면 규칙과 동일한 자격 판정
   ↓
오탐 방지 필터·0건 조기 종료
   ↓
[업무 시트] 기록 + 담당자 알림
```

### 기술 선택 이유

- n8n: 스케줄·DB·코드·시트·메일을 하나의 운영 흐름으로 관리
- Oracle SQL: 화면이 사용하는 복합 업무 규칙을 데이터 가까이에서 재현
- JavaScript 코드 노드: 코드값 변환, 날짜 처리, 출력 형식 정규화

### 직접 구현한 내용

- 화면 판정 규칙을 분석해 SQL로 재구성
- 환자별 최신 자격 결과 선택
- 외부 결과 미수신 대상 제외
- 기관 혼입과 중복 방지
- 특정 보험 유형 변환 규칙 표준화
- 날짜별 시트 생성·보존·정리
- 한국시간을 명시해 자정 경계 오류 방지
- 대상 없음 조기 종료와 오류 알림 분리

### 기술적 어려움과 해결 과정

초기에는 단순 비교 결과가 화면과 달라 거짓 불일치가 나타났다. 화면이 적용하는 선행조건과 예외 규칙을 다시 분해하고, 외부 결과 미수신·기관·최신 결과 기준을 추가했다. 서버 기본 시간대 때문에 날짜가 어긋나는 문제는 모든 날짜 계산에 한국시간을 명시해 해결했다.

### 보안 및 개인정보 보호

- DB 조회는 읽기 전용
- 환자정보는 승인된 업무 채널에만 전달
- 외부 포트폴리오에는 화면·SQL·데이터 예시를 포함하지 않음
- 공개 데모는 완전한 생성 데이터와 가상 자격 코드 사용

### 배포·운영·장애 대응

시트 정리 실패가 핵심 알림 흐름 전체를 중단하지 않도록 분리했고, 대상 없음은 정상 결과로 종료했다. 화면 결과와 자동화 결과를 교차 확인하는 검증 포인트를 문서화했다.

### 성과

매일 수기로 화면을 확인하던 업무를 자동 대상자 안내로 전환했다. 단순 데이터 추출이 아니라 현업 화면의 판단 규칙을 자동화에 반영해 실무에서 사용할 수 있는 정합성을 확보했다.

### 배운 점

업무자동화의 핵심은 도구가 아니라 업무 규칙을 정확하게 옮기는 일이다. 최신 데이터, 시간대, 기관 범위와 “데이터 없음”의 의미까지 설계해야 한다.

## 사례 3. 입퇴원요약 AI 초안 자동작성

### 프로젝트명

의료기록 기반 입퇴원요약 초안 생성 및 EMR 임시저장 자동화

### 기간과 진행 상태

- 기간: 2026.05~2026.07 개발·개선
- 상태: 운영 중·지속 개선

### 배경과 기존 문제

입퇴원요약과 초기 기록은 여러 임상기록을 확인해 항목별로 정리해야 한다. 단순 LLM 요약은 원문에 없는 내용을 만들거나, 서로 다른 항목이 섞이거나, 최종 확정 문서처럼 오해될 위험이 있었다.

### 요구사항

- 승인된 대상자의 원문 기록만 사용
- 항목별 초안 생성
- 원문이 없으면 임의 생성하지 않음
- EMR에는 서명 전 임시 초안으로 저장
- 의료진이 검토하고 최종 확정
- 적재 실패와 누락을 추적

### 담당 범위

대상 조회, 기록 병합, 항목별 프롬프트, 다중 LLM 흐름, 결과 정규화, EMR 임시저장, 감사로그와 실패 알림을 직접 구현했다.

### 시스템 구성과 데이터 흐름

```text
스케줄·수동 트리거
       ↓
[EMR 시스템] 대상·원문 조회
       ↓
긴 본문 분할 조회·재조립
       ↓
항목별 LLM 초안 생성
       ↓
병합·원문 없음 처리·품질 규칙
       ↓
[EMR 시스템] 임시저장
       ↓
감사로그·실패 알림
       ↓
의료진 검토·서명
```

### 기술 선택 이유

- n8n: DB 조회, 다중 LLM, 병합, API 적재, 감사 흐름을 시각적으로 운영
- 항목별 LLM 분리: 한 번의 긴 요약보다 항목 목적과 입력을 명확하게 통제
- 임시저장: AI 결과가 최종 의료기록으로 바로 확정되는 위험 방지

### 직접 구현한 내용

- 복수 트리거 입력 정규화
- 긴 의료기록의 조각 조회와 병합
- 항목별 프롬프트와 원문 충실 규칙
- 다중 분기 완료 후에만 다음 단계로 진행하는 병합 게이트
- 원문 없음 표현을 빈값으로 정규화
- EMR 임시저장 데이터 구성
- HTTP 응답과 업무 성공 값을 함께 확인
- 감사기록과 실패 알림

### 기술적 어려움과 해결 과정

긴 본문을 한 번에 조회하기 어려워 조각 단위로 읽고 순서대로 병합했다. 여러 LLM 분기가 먼저 끝나는 순서가 달라 일부 항목이 빠질 수 있어 모든 입력이 도착해야 다음 단계로 가는 동기화 게이트를 사용했다. API 응답 성공만으로 실제 저장 성공을 판단하지 않고 업무 결과 값을 추가 확인했다.

### 보안 및 개인정보 보호

- 실제 임상기록을 외부 모델 학습자료나 포트폴리오 예시로 사용하지 않음
- 승인된 내부 처리 범위와 수신자만 사용
- AI 결과는 초안이며 의료진 검토 전 확정되지 않음
- 공개 재구현은 공개 문서나 생성된 가상 기록만 사용

### 배포·운영·장애 대응

테스트 환경에 운영 스케줄이 활성화되지 않도록 배포 순서를 문서화했다. 참조 기록 없음, LLM 결과 없음, 저장 실패를 서로 다른 결과로 처리하고 감사·알림 흐름을 함께 운영했다.

### 성과

의료진이 원문을 확인하며 초안을 작성하던 과정에 AI 보조 흐름을 도입했다. AI가 최종 판단을 대신하지 않도록 임시저장과 인적 검토를 구조에 포함했다.

### 배운 점

의료 AX에서는 모델 성능만큼 입력 범위, 환각 방지, 승인 단계, 감사 가능성, 실패 시 중단 조건이 중요하다.

## 사례 4. DUR·개인투약이력 EMR API 개발

### 프로젝트명

EMR 화면 내부 기능을 재사용 가능한 내부 API로 확장

### 기간과 진행 상태

- 기간: 2026.07
- 상태: 운영 중

### 배경과 기존 문제

의약품 안전 점검과 개인투약이력 조회 기능이 EMR 화면 내부 호출에 묶여 있어 n8n이나 다른 내부 시스템에서 표준 방식으로 재사용하기 어려웠다.

### 요구사항

- 기존 화면 기능을 훼손하지 않고 외부 호출 인터페이스 제공
- 기존 비즈니스 서비스와 데이터 접근 계층 재사용
- 민감한 환자·인증정보가 로그에 남지 않도록 처리
- 컴포넌트·설정·매퍼의 배포 정합성 확보

### 담당 범위

EMR 측 Java 컴포넌트, 서비스 계층 연결, DAO·SQL 매퍼, 라우팅 설정과 배포 문서를 직접 개발했다.

### 시스템 구성과 데이터 흐름

```text
[내부 자동화·업무 시스템]
          ↓
      [내부 API]
          ↓
  EMR App·Manager·DAO
          ↓
기존 안전점검·이력조회 기능
          ↓
   표준 데이터셋 응답
```

### 기술 선택 이유

- 기존 EMR 프레임워크 계층 재사용: 화면 기능과 다른 별도 로직이 생기는 위험을 줄임
- Java·iBatis: 운영 중인 레거시 구조와 배포 체계에 맞춤
- JSON 직렬화 라이브러리 전환: 실제 classpath에서 사용할 수 있는 의존성 선택

### 직접 구현한 내용

- 외부 호출을 받는 EMR 컴포넌트와 메서드
- 기존 안전점검·조회 서비스 오케스트레이션
- 개인투약이력 조회용 DAO·SQL·매퍼
- 입력값 null·형식 검증
- 오류 응답과 감사 처리
- 설정·컴포넌트·매퍼·빌드 단위를 함께 반영하는 배포 순서

### 기술적 어려움과 해결 과정

레거시 컴포넌트는 코드 한 파일만 반영해서 동작하지 않고 설정·맵·SQL·빌드가 함께 맞아야 했다. 사용할 수 없는 라이브러리는 운영 classpath에 있는 대안으로 변경했고, 입력 누락과 null 흐름을 방어했다. 빌드 성공을 업무 성공으로 간주하지 않고 실제 호출과 데이터 결과를 확인하는 절차를 정리했다.

### 보안 및 개인정보 보호

- 환자 식별정보와 외부 조회 자격정보를 로그로 출력하지 않음
- 내부 API 주소·컴포넌트명·테이블명은 공개하지 않음
- GitHub 공개본은 가상 의료업무 API로 구조만 재구현

### 배포·운영·장애 대응

코드·설정·매퍼의 비대칭 반영을 막기 위해 하나의 배포 단위로 관리했다. 빌드 후 제한된 호출 검증과 실제 결과 재확인을 거쳐 운영 반영했다.

### 성과

화면 안에 있던 기능을 내부 자동화와 타 시스템에서 호출할 수 있는 형태로 확장했다. n8n 수준의 연동을 넘어 EMR 내부 계층을 직접 개발할 수 있는 역량을 보여준다.

### 배운 점

레거시 API 개발은 새 엔드포인트 작성만이 아니라 기존 업무 부수효과, 권한, 설정, 배포 순서와 실데이터 검증을 함께 다루는 일이다.

## 사례 5. 위수탁 수납 자동화 안전 재설계

### 프로젝트명

금전 업무 자동화의 이중 필터·전체 중단·모니터링 설계

### 기간과 진행 상태

- 기간: 2026.03~2026.06 재설계
- 상태: 운영 중

### 배경과 기존 문제

기관 간 검사비 수납을 반복 처리하는 업무를 자동화했으나 초기 구현에서 대상 판정 오류로 잘못된 수납이 발생했다. API가 성공 형태를 반환해도 실제 업무 결과가 기대와 다를 수 있었다.

### 요구사항

- 승인된 미수 유형만 처리
- 일부라도 금액 조건이 맞지 않으면 전체 중단
- 처리 후 실제 미처리 잔존 여부 확인
- 자동 취소처럼 위험한 기능은 범위에서 제외

### 담당 범위

n8n 워크플로우 개발, 장애 원인 분석, 복구 지원, 안전장치와 모니터링 재설계를 직접 수행했다.

### 시스템 구성과 데이터 흐름

```text
스케줄·수동 승인
      ↓
[EMR 시스템] 대상 조회
      ↓
서버 조건 + 클라이언트 이중 필터
      ↓
금액·부분수납 전체 중단 가드
      ↓
[내부 API] 수납 처리
      ↓
결과 판정 + 지연 모니터링
```

### 기술 선택 이유

- n8n: 처리 단계와 가드 조건을 명시적으로 분리
- 이중 필터: 한쪽 조건 변경이나 응답 이상이 잘못된 처리로 이어지는 위험 완화
- 전체 중단: 금전 자동화에서 부분 성공보다 검토 가능한 실패를 우선

### 직접 구현한 내용

- 조회·처리 API 연계
- 데이터 타입·빈 응답 정규화
- 허용 대상 이중 필터
- 잔액과 처리금액 불일치 시 전체 중단
- 업무 성공 값과 실제 잔존 대상 비교
- 처리 흐름과 별도의 모니터링
- 위험한 자동 취소 기능 제외 결정

### 기술적 어려움과 해결 과정

배치 처리 응답의 건수가 실제 변경된 건수를 의미하지 않는 구조가 있었다. 따라서 응답만으로 성공을 판단하지 않고 실제 데이터를 다시 조회하는 원칙을 도입했다. 장애를 코드 수정으로 끝내지 않고 유사 자동화에 적용할 공통 배포·검증 기준으로 확장했다.

### 보안 및 개인정보 보호

- 계좌·금액·대상자 정보 비공개
- 공개 문서에는 실제 코드값과 테이블 구조를 포함하지 않음
- 공개 데모는 가상 청구·수납 데이터만 사용

### 배포·운영·장애 대응

비활성 상태로 가져온 뒤 제한된 데이터로 검증하고, 운영 흐름과 모니터링을 함께 활성화했다. 오류·0건·중단을 서로 다른 알림으로 구분했다.

### 성과

반복 수납 업무를 자동화하는 동시에 장애 경험을 이중 필터, 전체 중단, 결과 재조회, 모니터링이라는 재발 방지 구조로 전환했다.

### 배운 점

금전·의료 업무 자동화에서는 처리량보다 잘못된 자동 처리를 멈추고 설명할 수 있는 구조가 우선이다.

## 사례 6. NHIC 자격조회 배치 서버 이관

### 프로젝트명

폐쇄망·레거시 제약을 해결한 자격조회 배치 운영 서버 이관

### 기간과 진행 상태

- 기간: 2026.03
- 상태: 이관 완료·운영

### 배경과 기존 문제

개발 PC에서 실행되던 자격조회 배치를 운영 서버로 옮겨 상시 운영해야 했다. 외부 연계용 32비트 COM, 폐쇄망 패키지 설치, 방화벽·DNS·프록시 제약 때문에 동일한 실행환경을 단순 복제할 수 없었다.

### 요구사항

- 32비트 구성요소 호환
- 인터넷이 없는 환경에서 의존성 설치
- 운영 서버에서 접근 가능한 데이터 경로 확보
- 실패 재시도와 스케줄 운영
- 개발 PC 의존 제거

### 담당 범위

Python 실행환경 구성, 오프라인 패키지 설치, COM 연동, 네트워크 경로 진단, 데이터소스의 내부 API 전환, 재시도 배치와 스케줄 등록을 직접 수행했다.

### 시스템 구성과 데이터 흐름

```text
Windows 작업 스케줄러
        ↓
  Python 32비트 배치
        ↓
 외부 자격조회 COM 연계
        ↓
[EMR 내부 API] 대상·결과 처리
        ↓
 실패 로그·재시도
```

### 기술 선택 이유

- 32비트 Python: 기존 COM 구성요소와의 호환
- 오프라인 패키지: 폐쇄망 운영 정책 준수
- EMR 내부 API: 직접 DB 연결이 차단된 환경에서 승인된 경로 사용
- 작업 스케줄러: Windows 운영환경의 표준 무인 실행 방식

### 직접 구현한 내용

- 32비트 실행환경과 경로 구성
- 오프라인 wheel·패키지 설치
- DB 직접 연결부터 내부 API까지 단계별 연결 진단
- API 요청·응답 구조에 맞춘 클라이언트 변경
- 실패 후 재시도 배치
- 작업 스케줄러 등록과 운영 문서화

### 기술적 어려움과 해결 과정

DB·도메인·프록시 경로마다 서로 다른 실패가 나타났다. 한 번에 우회하지 않고 DNS 해석, 포트 접근, 라우팅, API 매핑, 인증 단계를 분리해 확인했다. 최종적으로 운영 서버가 접근 가능한 내부 API 경로로 데이터소스를 전환했다.

### 보안 및 개인정보 보호

- 내부 주소·포트·기관 코드·인증값 비공개
- 패킷·로그 원본을 포트폴리오에 포함하지 않음
- 자격조회 결과와 환자 데이터 예시를 사용하지 않음

### 배포·운영·장애 대응

실패 시 일정 시간 후 재시도하고, 무인 실행 조건을 고려해 스케줄러를 구성했다. 이관 과정과 장애 진단 순서를 문서화해 재현 가능한 운영 절차로 남겼다.

### 성과

개발 PC에 의존하던 배치를 폐쇄망 운영 서버로 이관했다. 자동화 개발뿐 아니라 네트워크와 레거시 실행환경 문제를 해결해 실제 운영까지 연결하는 역량을 보여준다.

### 배운 점

현장 자동화는 코드가 동작하는 것과 운영환경에서 지속 실행되는 것이 다르다. 호환성, 네트워크, 인증, 재시도, 운영 문서까지 포함해야 배포가 완료된다.
